# MVP 범위 정의서

상태: 제품/운영 초안
기준 문서: `docs/service-overview.md`, `docs/prd.md`, `docs/privacy-verification-data-policy.md`, `docs/collaboration-log.md`
대상 서비스: 프리미엄 검증 기반 AI 소개팅 서비스

## 1. 목적

이 문서는 첫 MVP에서 반드시 만들 기능과 의도적으로 미룰 기능을 고정한다.
목표는 완성형 소개팅 플랫폼이 아니라, “검증된 사람을 안전하게 소개받는 경험”이 실제 수요와 운영 가능성을 갖는지 검증하는 것이다.

## 2. MVP 범위 원칙

- 무한 탐색형 서비스가 아니라 제한 추천형 서비스로 검증한다.
- 셀렉트/프리미엄 회원은 강한 검증 기준을 통과한 회원으로 다룬다.
- 일반 회원은 셀렉트 회원 목록을 직접 볼 수 없다.
- 일반 회원은 명시적으로 opt-in한 경우에만 셀렉트 회원에게 추천 후보로 들어간다.
- AI는 최종 매칭 판단자가 아니라 추천 이유와 조율 포인트 설명을 담당한다.
- 초기 추천 품질은 운영자 큐레이션과 룰 기반 후보 필터링으로 보완한다.
- 인증자료 원본은 장기 보관하지 않고, DB에는 민감 원문을 저장하지 않는다.
- 사용자 앱은 `React + Vite + TypeScript + Capacitor` 단일 코드베이스를 우선한다.

## 3. MVP 포함 범위

| 영역 | 포함 기능 | 목적 |
|---|---|---|
| 계정 | 가입, 로그인, 계정 상태 | 추천과 검증 신청의 기본 권한을 관리한다. |
| 프로필 | 기본 프로필, 사진, 선호 조건, 공개 범위 | 추천 가능 상태를 만든다. |
| 설문 | 성향/가치관 설문, 수정 | AI 추천 feature의 최소 입력을 만든다. |
| 일반 회원 opt-in | 셀렉트 추천 후보 참여 동의/철회 | 일반 회원의 노출 통제와 동의를 보장한다. |
| 셀렉트 검증 | 검증 신청, 자료 업로드, 상태 조회 | 강한 검증 모델의 수요와 운영 난이도를 확인한다. |
| 관리자 심사 | 심사 큐, 열람 사유, 승인/반려/재제출 | 검증 품질과 감사 가능성을 확보한다. |
| 추천 후보군 | 등급, 검증 상태, opt-in, 차단/신고 기반 필터링 | 추천 가능한 후보를 안전하게 좁힌다. |
| 추천 카드 | 제한된 카드, 추천 이유, 수락/거절/숨김/신고 | 핵심 사용자 경험을 검증한다. |
| AI 추천 이유 | 가능성 중심 설명, 주의점, confidence, source_version | 설명 가능한 매칭의 가치를 검증한다. |
| 안전 | 신고, 차단, 운영자 처리 | 초기 신뢰와 안전성을 확보한다. |
| 피드백 | 추천 품질/설명 정확도 피드백 | 다음 추천 개선 데이터를 모은다. |

## 4. MVP 제외 범위

| 제외 항목 | 제외 이유 |
|---|---|
| 실시간 채팅 고도화 | 매칭 품질 검증 전에는 범위가 커진다. |
| 복잡한 결제/구독/포인트/쿠폰 | 과금 실험은 검증 경험이 안정된 뒤 분리한다. |
| 외부 SNS/재직/소득 자동 검증 전체 | 초기에는 운영자 심사로 기준과 리스크를 먼저 확인한다. |
| 고도화 추천 엔진 전체 | MVP는 룰 기반 후보군과 설명 생성 중심으로 시작한다. |
| 대규모 소셜 피드 | 핵심 가치인 검증 추천과 직접 관련이 낮다. |
| 고급 AI 연애 코칭/데이트 플랜 | 추천 이유의 가치 검증 후 확장한다. |
| 다국가/다언어 운영 | 개인정보, 결제, 약관 리스크가 커진다. |
| 셀렉트 회원 공개 탐색 | 프리미엄 풀 보호 원칙과 충돌한다. |
| 완성형 관리자 BI/통계 대시보드 | 초기에는 심사, 신고, 감사 로그 중심으로 제한한다. |

## 5. 단계별 릴리즈 범위

### MVP-0 내부 운영 프로토타입

목적: 운영자가 검증 심사와 추천 큐레이션을 실제로 수행할 수 있는지 확인한다.

포함:

- 관리자 로그인과 역할 구분
- 검증 신청 더미/테스트 데이터 심사
- 인증자료 참조, 열람 사유, 감사 로그
- 수동 추천 후보 큐레이션
- 추천 카드 미리보기

완료 기준:

- 심사자가 승인/반려/재제출 요청을 처리할 수 있다.
- 열람 사유와 감사 로그가 남는다.
- 인증자료 원본 저장 금지/제한 보관 원칙을 구현 설계로 설명할 수 있다.

### MVP-1 클로즈드 베타

목적: 실제 사용자가 검증, opt-in, 추천 카드, AI 추천 이유를 이해하고 사용하는지 확인한다.

포함:

- 가입/로그인
- 기본 프로필/사진/설문
- 셀렉트 추천 opt-in과 철회
- 셀렉트 검증 신청과 상태 조회
- 제한된 추천 카드
- AI 추천 이유
- 수락/거절/숨김/신고
- 운영자 심사와 큐레이션

완료 기준:

- 일반 회원이 셀렉트 추천 opt-in 의미를 이해하고 선택할 수 있다.
- 셀렉트 회원은 직접 공개 노출 없이 추천을 받을 수 있다.
- 추천 카드 수락/거절/숨김이 중복 처리되지 않는다.
- AI 추천 이유에 민감 원문이나 우열 표현이 포함되지 않는다.

### MVP-2 공개 베타 후보

목적: 클로즈드 베타 결과를 바탕으로 유료화와 자동화 범위를 정한다.

후보 포함:

- 검증 심사비 또는 셀렉트 멤버십 결제 실험
- 추천 빈도/품질 조정
- 인기 일반 회원 선별 기준
- 제한적 대화 연결
- 관리자 통계 대시보드

MVP-2는 MVP-1 지표 확인 전에는 확정하지 않는다.

## 6. PRD 기능별 MVP 판정

| PRD ID | 기능 | MVP 판정 | 비고 |
|---|---|---|---|
| F-001 | 가입 및 로그인 | 포함 | 모든 흐름의 전제 조건이다. |
| F-002 | 기본 프로필 작성 | 포함 | 추천 가능 상태를 만든다. |
| F-003 | 사진 등록 | 포함 | 추천 카드 품질에 필요하다. |
| F-004 | 성향/가치관 설문 | 포함 | AI 추천 feature의 핵심 입력이다. |
| F-005 | 일반 회원 셀렉트 추천 opt-in | 포함 | 프리미엄 풀 보호와 일반 회원 동의를 동시에 보장한다. |
| F-006 | 셀렉트/프리미엄 검증 신청 | 포함 | 서비스 차별점 검증의 핵심이다. |
| F-007 | 관리자 심사 | 포함 | 초기 자동 검증 대신 운영 품질을 확보한다. |
| F-008 | 추천 후보군 생성 | 포함 | 룰 기반 필터링부터 시작한다. |
| F-009 | 추천 카드 | 포함 | 핵심 사용자 경험이다. |
| F-010 | AI 추천 이유 | 포함 | 설명 가능한 매칭의 가치를 검증한다. |
| F-011 | 신고/차단 | 포함 | 안전 기능이 없으면 베타 운영이 어렵다. |
| F-012 | 피드백 수집 | 포함 | 추천 품질 개선과 운영 판단에 필요하다. |
| A-001~A-005 | 관리자 심사 기본 기능 | 포함 | 검증 운영의 필수 기능이다. |
| A-006 | 추천 큐레이션 후보 확인 | 포함 | 완전 자동 추천 전까지 필요하다. |
| A-007 | 신고/차단/제재 처리 | 포함 | 안전 운영의 필수 기능이다. |
| A-008 | 감사 로그 조회 | 포함 | 민감자료 열람 통제에 필요하다. |
| A-009 | 기본 지표 대시보드 | 축소 포함 | 상세 BI가 아니라 최소 운영 지표만 포함한다. |

## 7. 릴리즈 전 필수 결정

MVP 구현 착수 전에 다음 문서 또는 정책을 먼저 확정해야 한다.

1. 회원 등급 정책서
2. 셀렉트/프리미엄 검증 기준표
3. 인증자료 수집/검증/보관 명세서
4. 노출 정책서
5. 매칭 정책서
6. 사용자 플로우와 화면 명세서
7. API/DB 설계 초안
8. 관리자 심사/운영 정책서

## 8. MVP 성공 기준

| 지표 | 1차 판단 기준 |
|---|---|
| 프로필 완성률 | 가입 후 추천 가능 상태까지 도달하는 비율 |
| 설문 완료율 | AI 추천 준비도 |
| 셀렉트 검증 신청률 | 강한 검증 모델의 수요 |
| 검증 승인/반려 비율 | 검증 기준의 현실성 |
| 일반 회원 opt-in 비율 | 셀렉트 후보 참여 모델 수용도 |
| 추천 카드 수락률 | 큐레이션 품질 |
| AI 추천 이유 열람률 | 설명 기능의 가치 |
| 신고/차단 비율 | 안전성 리스크 |
| 재방문율 | 제한 추천 방식의 지속성 |

1. MVP-1 포함 범위가 아직 큰지, 1차 구현에서 더 줄일 항목이 있는지

5. 추천 후보군 생성과 AI 추천 이유의 최소 API/DB 경계
6. Capacitor 앱에서 사진/인증자료 업로드, 로컬 잔존 방지, 중복 액션 방지 구현 리스크
