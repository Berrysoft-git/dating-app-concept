# PRD 제품 요구사항 문서

상태: 제품/운영 초안  
기준 문서: `docs/service-overview.md`, `docs/product-concept.md`, `docs/collaboration-log.md`, `docs/privacy-verification-data-policy.md`  
대상 서비스: 프리미엄 검증 기반 AI 소개팅 서비스

## 1. 목적

이 PRD는 MVP에서 실제로 구현할 제품 범위, 사용자 흐름, 기능 요구사항, 성공 기준을 정의한다.  
서비스의 핵심은 “검증된 사람을 안전하게 소개받는 경험”이며, 초기 버전은 완전 자동 매칭보다 검증, 큐레이션, 설명 가능한 추천의 품질 검증에 집중한다.

## 2. 배경

기존 소개팅 앱은 사진과 조건 중심 탐색에 치우쳐 있고, 사용자가 직접 많은 프로필을 훑어야 한다. 고신뢰 회원은 무분별한 노출을 부담스러워하고, 일반 회원은 좋은 상대를 만날 기회를 명확히 이해하기 어렵다.

이 서비스는 다음 방향으로 차별화한다.

- 셀렉트/프리미엄 회원은 강한 검증 기준을 통과한 회원으로 정의한다.
- 일반 회원은 셀렉트 회원 목록을 직접 조회하지 못한다.
- 일반 회원은 opt-in한 경우에만 셀렉트 회원에게 추천 후보로 들어간다.
- AI는 최종 판단자가 아니라 추천 이유, 궁합 요약, 조율 포인트 설명을 담당한다.
- 운영자는 초기 큐레이션 품질과 심사 일관성을 관리한다.

## 3. 목표

### 제품 목표

- 사용자가 제한된 고품질 추천을 신뢰하고 수락할 수 있게 한다.
- 셀렉트/프리미엄 회원의 노출 통제와 희소성을 보장한다.
- 일반 회원에게 셀렉트 추천 opt-in의 의미와 철회 가능성을 명확히 고지한다.
- 운영자가 검증 심사와 추천 큐레이션을 일관되게 수행할 수 있게 한다.
- AI 추천 이유가 사용자의 이해와 수락 결정에 도움을 주는지 검증한다.

### 사업 목표

- 셀렉트 검증 모델에 실제 수요가 있는지 확인한다.
- 검증 심사, 셀렉트 멤버십, AI 분석 리포트의 유료화 가능성을 검증한다.
- 초기에는 과도한 자동화보다 운영 가능한 고품질 매칭 경험을 만든다.

## 4. 비목표

MVP에서는 다음을 만들지 않는다.

- 완전 자동 고도화 매칭 엔진
- 복잡한 구독/포인트/쿠폰 경제 전체
- 실시간 채팅 고도화
- 외부 SNS 자동 검증 전체
- 다국가/다언어 운영
- 대규모 소셜 피드
- 고급 AI 연애 코칭 또는 데이트 플랜 생성
- 법무 검토가 끝난 최종 약관/개인정보처리방침

## 5. 사용자 유형

| 사용자 | 설명 | 핵심 목표 |
|---|---|---|
| 일반 회원 | 기본 가입 회원 | 안전하게 프로필을 만들고 추천을 받는다. |
| 인기 일반 회원 | 활동 품질과 신뢰 지표가 좋은 일반 회원 | 더 높은 품질의 추천 후보가 된다. |
| 셀렉트/프리미엄 회원 | 강한 검증을 통과한 회원 | 노출을 통제하면서 고품질 추천을 받는다. |
| 심사자 | 검증 신청을 심사하는 운영자 | 일관된 기준으로 승인/반려한다. |
| 운영자 | 추천 큐레이션과 신고 대응 담당 | 추천 품질과 안전성을 관리한다. |
| 보안 관리자 | 민감자료 접근과 감사 로그 관리 | 오남용과 정보 유출을 방지한다. |

## 7. MVP 기능 요구사항

### F-001 가입 및 로그인

우선순위: 필수

요구사항:

- 사용자는 계정을 생성하고 로그인할 수 있어야 한다.
- 계정 상태는 활성, 심사 대기, 제한, 탈퇴 예정, 탈퇴 완료로 구분한다.
- 로그인 후 사용자 권한과 검증 상태를 조회할 수 있어야 한다.

수용 기준:

- 신규 사용자가 가입 후 기본 프로필 작성 화면으로 이동한다.
- 제한된 사용자는 추천 카드와 검증 신청 기능에 접근할 수 없다.

### F-002 기본 프로필 작성

우선순위: 필수

요구사항:

- 사용자는 닉네임, 나이대, 지역 단위, 자기소개, 선호 조건을 입력할 수 있어야 한다.
- 필수 항목을 모두 채워야 추천 가능 상태가 된다.
- 사용자는 프로필 공개 범위와 추천 노출 범위를 설정할 수 있어야 한다.

수용 기준:

- 필수 항목 누락 시 추천 후보에 포함되지 않는다.
- 사용자는 언제든 프로필을 수정할 수 있다.

### F-003 사진 등록

우선순위: 필수

요구사항:

- 사용자는 프로필 사진을 등록할 수 있어야 한다.
- 사진은 관리자 또는 자동 정책에 의해 부적합 상태가 될 수 있다.
- 모바일 앱에서는 Capacitor 공식 플러그인을 통해 카메라/갤러리 접근을 처리한다.

수용 기준:

- 사진 업로드 실패 시 재시도할 수 있다.
- 부적합 사진은 추천 카드에 노출되지 않는다.

### F-004 성향/가치관 설문

우선순위: 필수

요구사항:

- 사용자는 성향, 관계관, 생활 패턴, 대화 스타일, 선호 조건 설문에 응답한다.
- 설문 응답은 AI 추천 feature로 변환된다.
- 민감 원문은 AI에 직접 전달하지 않고 요약 feature 중심으로 사용한다.

수용 기준:

- 최소 설문 완료 기준을 충족해야 추천 가능 상태가 된다.
- 사용자는 설문을 다시 수정할 수 있다.

### F-005 일반 회원 셀렉트 추천 opt-in

우선순위: 필수

요구사항:

- 일반 회원은 셀렉트 회원에게 추천 후보로 포함될지 직접 선택할 수 있어야 한다.
- opt-in 전에는 노출 범위, 철회 가능성, 셀렉트 회원 목록 비공개 원칙을 명확히 고지한다.
- opt-in 철회 후에는 신규 셀렉트 추천 후보군에 포함되지 않는다.

수용 기준:

- opt-in하지 않은 일반 회원은 셀렉트 회원 추천 후보에 들어가지 않는다.
- opt-in 철회 UI가 명확히 제공된다.

### F-006 셀렉트/프리미엄 검증 신청

우선순위: 필수

요구사항:

- 사용자는 셀렉트 검증을 신청할 수 있어야 한다.
- 검증 유형은 직업, 소득/자산, 전문 자격, 학력/경력, 공개 영향력 등으로 구분한다.
- 인증자료 원본은 별도 보안 저장소에 제한 기간만 보관한다.
- DB에는 원본 파일이나 고유식별번호 원문을 저장하지 않는다.

수용 기준:

- 사용자는 검증 상태를 대기, 승인, 반려, 재제출 요청으로 확인할 수 있다.
- 사용자는 심사 전 신청을 철회할 수 있다.

### F-007 관리자 심사

우선순위: 필수

요구사항:

- 심사자는 검증 신청 목록을 확인할 수 있어야 한다.
- 인증자료 열람 전 열람 사유 입력이 필요하다.
- 심사자는 승인, 반려, 재제출 요청을 처리할 수 있다.
- 모든 열람, 승인, 반려, 권한 변경은 감사 로그로 남는다.

수용 기준:

- 승인된 사용자는 검증 뱃지를 받는다.
- 반려된 사용자는 사유 코드와 재제출 가능 여부를 확인한다.
- 감사 로그는 운영자가 임의 수정할 수 없다.

### F-008 추천 후보군 생성

우선순위: 필수

요구사항:

- 후보군은 회원 등급, 검증 상태, opt-in 여부, 차단/신고 상태, 선호 조건을 반영해 생성한다.
- 일반 회원은 셀렉트 회원 목록을 직접 조회하지 못한다.
- 셀렉트 회원은 셀렉트 풀 추천과 opt-in 일반 회원 추천을 구분해 받을 수 있어야 한다.

수용 기준:

- 차단한 사용자나 신고 처리 중인 사용자는 추천 후보에서 제외된다.
- opt-in하지 않은 일반 회원은 셀렉트 후보군에 포함되지 않는다.

### F-009 추천 카드

우선순위: 필수

요구사항:

- 사용자는 제한된 수의 추천 카드를 받는다.
- 추천 카드에는 공개 가능한 프로필 정보와 추천 이유 요약이 포함된다.
- 사용자는 수락, 거절, 숨김, 신고를 선택할 수 있다.
- 수락/거절/숨김 액션은 중복 처리되지 않아야 한다.

수용 기준:

- 같은 추천 카드에 대해 중복 수락/거절이 발생하지 않는다.
- 숨김 처리한 후보는 일정 기간 다시 노출되지 않는다.

### F-010 AI 추천 이유

우선순위: 필수

요구사항:

- AI는 추천 이유, 잘 맞을 가능성, 초반 조율 포인트를 설명한다.
- 설명은 확정적 판단이 아니라 가능성 중심으로 작성한다.
- 소득, 직업, 외모, 학력의 우열 표현을 사용하지 않는다.
- AI 설명에는 인증자료 원본이나 민감 식별정보를 포함하지 않는다.

수용 기준:

- 추천 설명에는 `reason`, `caution`, `confidence`, `source_version`에 해당하는 구조가 있어야 한다.
- 사용자가 불쾌하거나 부정확한 설명을 신고할 수 있어야 한다.

### F-011 신고/차단

우선순위: 필수

요구사항:

- 사용자는 추천 카드, 프로필, 대화 관련 문제를 신고할 수 있어야 한다.
- 사용자는 특정 사용자를 차단할 수 있어야 한다.
- 지인 차단은 연락처 원문이 아니라 해시 기반으로 처리한다.

수용 기준:

- 신고된 사용자는 운영자 검토 대상에 들어간다.
- 차단된 사용자는 추천 후보에서 제외된다.

### F-012 피드백 수집

우선순위: 필수

요구사항:

- 사용자는 추천 품질, 설명 정확도, 불편 사유를 피드백할 수 있어야 한다.
- 피드백은 추천 품질 개선과 운영자 큐레이션 개선에 사용한다.

수용 기준:

- 수락/거절 이후 선택형 피드백을 남길 수 있다.
- 신고와 일반 피드백은 분리해서 처리한다.

## 9. 비기능 요구사항

### 보안

- 인증자료 원본은 별도 보안 저장소에 보관한다.
- 원본 파일은 제한 기간 이후 자동 삭제한다.
- 관리자 열람은 역할 기반 접근 제어와 감사 로그를 적용한다.
- DB에는 고유식별번호 원문, 인증자료 원본, 연락처 원문 목록을 저장하지 않는다.

### 개인정보

- 사용자에게 수집 목적, 보관 기간, 열람 주체, 파기 기준을 고지한다.
- opt-in과 검증 신청은 철회 가능해야 한다.
- AI 추천에는 비식별 feature를 사용한다.

### 프론트엔드/Capacitor

- 사용자 앱은 `React + Vite + TypeScript + Capacitor`를 기본으로 한다.
- 표준 HTML DOM을 유지한다.
- 화면 전환과 카드 애니메이션은 `transform`과 `opacity` 중심으로 구현한다.
- 카메라, 파일, 푸시, 로컬 저장소는 Capacitor 공식 플러그인을 래핑해서 사용한다.
- 인증자료가 WebView local storage, cache, 임시 파일에 남지 않도록 설계한다.

### 신뢰성

- 추천 카드 액션은 idempotency key를 사용한다.
- 파일 업로드 실패와 삭제 실패는 재시도 가능해야 한다.
- 운영자 작업은 감사 로그로 추적 가능해야 한다.

## 11. 주요 리스크

|---|---|

| 일반 회원이 셀렉트 구조를 박탈감으로 느낄 수 있음 | opt-in, 철회, 추천 기회 구조를 명확히 설명한다. |

| AI 설명이 부정확하거나 민감할 수 있음 | 가능성 중심 표현, 신고/피드백, source_version 기록을 적용한다. |
| 운영자 권한 오남용 | 권한 분리, 열람 사유, 감사 로그를 강제한다. |
| MVP 범위 과다 | 채팅 고도화, 결제 상품 전체, 자동 검증 전체는 제외한다. |
