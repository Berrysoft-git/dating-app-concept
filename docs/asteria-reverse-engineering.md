# Asteria 역기획서

이 문서는 기존 소개팅 앱 저장소 `26_A_Asteria`를 분석해 공개 가능한 수준으로 정리한 역기획서입니다. 실제 코드, 토큰, DB 접속정보, 샘플 개인정보, 내부 로그 원문은 포함하지 않습니다.

## 1. 서비스 정체성

Asteria의 원형은 단순 스와이프형 소개팅 앱보다 "검증된 회원을 시간대별로 큐레이션하는 프리미엄 매칭 서비스"에 가깝다.

핵심 방향은 다음과 같다.

- 무제한 탐색보다 제한된 시간대의 선별 추천을 강조한다.
- 일반 회원, 능력 인증 회원, 외모 인증 회원, 프리미엄 구독 회원을 나누어 신뢰와 희소성을 만든다.
- 직업, 소득, 자산, 학력, 인플루언서 등 강한 인증 뱃지를 매칭과 노출의 핵심 신호로 사용한다.
- 추천, 호감, 프로필 해금, 조건 검색, 쿠폰, 골드 재화가 결합된 수익 구조를 가진다.
- 가입 심사와 인증 심사는 관리자 운영 도구를 전제로 설계되어 있다.

## 2. 기술 구조

분석된 구현은 세 축으로 나뉜다.

| 영역 | 구현 | 역할 |
| --- | --- | --- |
| 모바일 앱 | Flutter, Riverpod, GoRouter, Dio, Secure Storage, Firebase Messaging | 사용자 앱, 온보딩, 추천, 활동, 매칭, 마이페이지 |
| API 서버 | ASP.NET Core API, EF Core, MariaDB, JWT, SignalR | 인증, 프로필, 매칭, 채팅, 지갑, 관리자 API |
| 웹/관리 | Blazor WebAssembly/Razor | 관리자 대시보드, 웹 가입/프로필/지갑/채팅 화면 |

구현상 앱 이름은 `Asteria`, 내부 솔루션/문서명은 `Project Rich`가 혼재한다. 제품화 전에는 브랜드명, 앱 패키지명, API 네임스페이스, 문서명을 하나로 정리해야 한다.

## 3. 사용자 여정

### 가입과 인증

1. 스플래시에서 인증 상태를 확인한다.
2. 전화번호 인증 또는 SNS 로그인을 선택한다.
3. 신규 회원은 기본 정보, 상세 정보, 사진 등록을 진행한다.
4. 인증 선택 단계에서 일반, 능력 인증, 외모 인증 중 하나를 선택한다.
5. 능력 인증은 서류 업로드, 외모 인증은 사진 업로드를 거친다.
6. 사용자는 심사 대기 화면으로 이동하고, 관리자는 대시보드에서 승인/반려한다.

### 메인 사용

앱의 하단 구조는 발견, 추천, 활동, 인연, 내 정보 흐름으로 구성되어 있다.

- 발견: 오늘의 카드와 프로필 탐색
- 추천: 조건 기반 추천, 월드컵형 선호 수집, 추천 결과
- 활동: 보낸 호감, 받은 호감, 방문/활동 히스토리
- 인연: 매칭 목록과 연락처 상세
- 내 정보: 프로필 수정, 사진 관리, 인증 센터, 지갑, 쿠폰, 설정, 고객지원

### 운영자 여정

관리자는 별도 웹 대시보드에서 다음 작업을 수행한다.

- 심사 대기 회원 목록 확인
- 회원 상세, 사진, 인증 서류, 골드 이력 확인
- 회원 승인/반려, 반려 사유 선택
- 능력 인증 서류 승인/반려
- 쿠폰 발급과 사용 현황 관리
- 골드 지급
- 감사 로그 확인

## 4. 핵심 기능 역분해

### 인증과 회원 등급

Asteria는 단순 본인 인증보다 강한 "능력/외모/프리미엄" 신호를 설계했다.

- `VerificationTier`: 기본, 능력 인증, 외모 인증으로 구분된다.
- `UserAbilities`: 사용자가 제출한 능력 인증 항목과 증빙 파일 상태를 저장한다.
- `MasterAbilities`: 인증 항목 마스터다. 자산, 직업/학력, 매력/영향력 계열로 나뉜다.
- 승인된 인증은 프로필 뱃지와 추천 필터의 핵심 기준이 된다.

이 구조는 새 기획의 `셀렉트/프리미엄 검증 회원`으로 확장하기 좋다. 다만 새 기획에서는 "유료 구독자"와 "강한 자격 검증 회원"을 분리해야 한다.

### 매칭과 큐레이션

매칭은 타임 슬롯 중심으로 설계되어 있다.

| 슬롯 | 기획상 의미 | 구현 방향 |
| --- | --- | --- |
| 08시 | 활동성 추천 | 최근 접속 우선 |
| 11시 | 근거리/지역 | 같은 지역 우선 |
| 14시 | 프리미엄 추가 카드 | 현재는 랜덤 성격 |
| 17시 | 라이프스타일 | 직업군, MBTI 유사도 |
| 20시 | AI 프라임 추천 | 현재는 랜덤 성격 |
| 23시 | Midnight Choice | 인증 등급 우선 |

서버는 새벽 배치로 `PendingMatchQueue`를 만들고, 사용자가 카드 조회 시점에 현재 시간대까지의 큐를 `DailyIntroductions`로 이동한다. 차단 회원과 연락처 기반 지인 차단은 후보군에서 제외한다.

현재 코드는 "AI 점수"를 실제로 계산하지 않고 `100.0` 같은 자리표시자와 랜덤/룰 기반 선택을 사용한다. 새 기획에서는 이 구조를 유지하되 AI 분석 점수, 추천 이유, 리스크 설명을 별도 엔진으로 붙이는 것이 현실적이다.

### 프로필과 노출 제어

프로필은 기본 정보, 사진, 직업군, 키, 체형, MBTI, 가치관 JSON, 이상형 JSON, 인증 뱃지를 가진다.

프로필 조회는 난독화된 사용자 해시를 사용하고, 프로필 해금/데일리 카드 액션 여부에 따라 일부 정보를 제한한다. 이 설계는 새 기획의 "일반 회원은 프리미엄 풀을 직접 열람하지 못한다"는 원칙으로 확장할 수 있다.

새 기획에 맞춘 노출 정책은 다음처럼 정리하는 것이 좋다.

- 일반 회원은 셀렉트 회원 목록을 직접 검색하지 못한다.
- 일반 회원은 본인이 opt-in한 경우에만 셀렉트 회원에게 추천 후보로 들어간다.
- 셀렉트 회원은 셀렉트 풀 내부 추천과 opt-in 일반 회원 추천을 분리해서 받는다.
- 운영자는 민감한 인증 자료가 아니라 승인 결과와 뱃지만 장기 보관한다.

### 재화와 수익 모델

Asteria는 골드 기반 재화를 중심으로 기능을 묶었다.

- 지갑: 유료 골드와 보너스 골드를 분리한다.
- 쿠폰: 골드, 아이템, 티켓 지급을 지원한다.
- 프로필 해금: 특정 후보를 골드로 열람한다.
- 랜덤 뽑기/조건 추천: 등급과 조건에 따라 비용을 다르게 책정한다.
- 아이템: 호감 강조, 추천권, 티켓형 상품으로 확장 가능하다.

새 기획에서는 단순 과금형 뽑기보다 "검증 심사비, 셀렉트 멤버십, 운영자 큐레이션, AI 분석 리포트" 중심으로 수익 구조를 재정렬하는 편이 프리미엄 포지션에 맞다.

### 안전과 프라이버시

구현과 문서에서 확인되는 안전 장치는 다음과 같다.

- 전화번호 해시와 서버 pepper 기반 중복/차단 처리
- 연락처 기반 지인 차단
- JWT 인증과 권한 기반 관리자 API
- 신고/차단/문의/공지 기능
- SignalR 알림
- 채팅 읽음 처리와 채팅방 권한 검사

다만 실제 제품 수준으로 보려면 추가 검증이 필요하다.

- PASS/CI/DI 실명 인증은 설계 비중이 크지만 구현상 완성도가 낮다.
- 문서에는 서류 심사 후 원본 파기를 전제로 하지만 코드 차원에서 즉시 파기 보장이 명확하지 않다.
- 캡처 방지, 미디어 자동 파기, 유해 이미지 필터링은 기획/모델 중심이고 완결 구현은 아니다.
- 개발/테스트용 우회 엔드포인트와 mock 데이터가 남아 있어 운영 전 보안 정리가 필요하다.

## 5. 데이터 모델 요약

주요 테이블은 다음 묶음으로 볼 수 있다.

- 계정: `Users`, `SystemAdmins`, `SystemActionLogs`
- 프로필: `UserProfiles`, `UserPhotos`, `TypeMBTI`, `TypeBDSM`, `TypeJobCategory`
- 인증: `UserAbilities`, `MasterAbilities`, `MasterRejectionReasons`
- 매칭: `PendingMatchQueue`, `DailyIntroductions`, `ProfileUnlocks`, `UserMatches`
- 채팅/알림: `ChatRooms`, `ChatMessages`, `Notifications`
- 재화: `UserWallet`, `GoldLogs`, `Coupons`, `UserCouponUsages`, `MasterItems`, `UserItems`
- 안전/지원: `UserBlocks`, `SystemNotices`, `UserInquiries`

새 기획에서 추가하거나 분리할 데이터는 다음이다.

- 셀렉트 검증 신청서와 심사 결과
- 인증 원본 파일의 보관/파기 이력
- 일반 회원의 셀렉트 추천 opt-in 설정
- 추천 이유와 AI 분석 리포트
- 추천 거절 사유와 피드백
- 셀렉트 회원에게 노출된 일반 회원의 노출 이력
- 운영자 큐레이션 큐와 수동 개입 로그

## 6. 현재 구현 상태 판단

구현은 "기획이 넓고, 핵심 골격은 잡혀 있지만, 서비스 품질을 결정하는 AI/검증/보안 자동화는 아직 MVP 이전" 상태다.

완성도가 비교적 높은 축은 다음이다.

- 전화번호/SNS 로그인 골격
- 프로필/사진/인증 제출 흐름
- 관리자 승인/반려 흐름
- 시간대 매칭 큐와 데일리 카드 API
- 골드 지갑, 쿠폰, 조건 추천 과금 흐름
- 차단/지인 차단 흐름
- 채팅방/메시지 기본 API

제품화 전 보완이 필요한 축은 다음이다.

- AI 매칭 점수와 설명 생성
- 실명/PASS/CI-DI 중복 방지
- 인증 서류 원본 파기와 감사 가능성
- 결제 검증과 환불/정산
- 미디어 보안, 캡처 방지, 신고 후 열람 정책
- 관리자 위험 엔드포인트 잠금
- mock 데이터, 테스트 번호, 로컬 URL, 빌드 산출물 정리
- Flutter 앱과 Blazor 웹의 역할 중복 정리

## 7. 새 기획에 반영할 결론

Asteria에서 가져갈 핵심은 "시간대별 큐레이션, 강한 인증 뱃지, 관리자 심사, 지인 차단, 골드/쿠폰 운영"이다.

새 기획에서 바꿔야 할 핵심은 다음이다.

- `프리미엄`을 단순 유료 구독이 아니라 `셀렉트 검증 회원`으로 재정의한다.
- 일반 회원이 셀렉트 회원을 직접 보는 구조를 제거한다.
- 일반 회원은 opt-in 기반으로만 셀렉트 회원에게 추천될 수 있게 한다.
- AI는 후보를 고르는 기능보다 "추천 이유와 관계 리스크를 설명하는 기능"부터 MVP에 넣는다.
- 운영자 큐레이션을 초기에 유지해 매칭 품질을 보정한다.
- 인증 원본은 장기 보관하지 않고 심사 결과, 뱃지, 감사 로그만 남긴다.

## 8. 재기획 MVP

1차 MVP는 다음 순서가 적합하다.

1. 단일 웹 앱으로 가입, 프로필, 인증 신청, 추천 카드, 수락/거절을 구현한다.
2. 사용자 앱은 HTML/CSS/JS 기반으로 만들고 Capacitor로 모바일 패키징한다.
3. API와 관리자 구조는 Asteria의 ASP.NET Core/관리자 대시보드 설계를 참고한다.
4. 매칭은 처음부터 완전 자동화하지 않고 운영자 큐레이션과 룰 기반 점수를 섞는다.
5. AI는 성향 설문 요약, 후보 추천 이유, 주의할 차이점 설명부터 붙인다.
6. 셀렉트 검증은 신청, 증빙 업로드, 관리자 심사, 원본 파기, 뱃지 발급으로 제한해 시작한다.

## 9. 공개 기획서 반영 요약

이 역기획 결과를 기존 제품 콘셉트에 반영할 때의 핵심 문장은 다음이다.

> Asteria의 기존 구현은 프리미엄 소개팅 서비스의 기술 골격을 이미 갖고 있다. 새 서비스는 이 골격에서 "강한 검증 회원", "비대칭 노출", "AI 설명형 큐레이션", "운영자 품질 관리"만 명확히 재정의하면 더 선명한 프리미엄 매칭 서비스로 발전할 수 있다.
