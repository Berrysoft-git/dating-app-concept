# 협업 로그

이 문서는 Discord 멘션 기반 협업에서 논의한 작업 상태, 합의된 기준, 열린 쟁점을 정리한다. 제품 콘셉트 문서는 합의된 내용만 반영하고, 아직 결정되지 않은 내용은 이 문서의 쟁점 단위로 남긴다.

## Discord 멘션 기반 협업 규칙

1. 초안 담당자가 문서를 작성하거나 수정한다.
2. 초안 담당자는 Discord에서 리뷰어를 멘션해 리뷰를 요청한다.
3. 리뷰어는 담당 관점에서 구체적인 쟁점만 남긴다.
4. 이견은 본문에 섞지 않고 별도 쟁점으로 분리한다.
5. 합의되면 상태를 합의 완료로 표시한다.
6. 합의된 내용만 GitHub 문서에 반영한다.
7. 민감한 운영 정보, 사용자 개인정보, 인증자료, 토큰, 내부 로그는 공개 문서에 남기지 않는다.

### 제품/운영 관점 결정 초안

- 인증자료 원본은 별도 보안 저장소에 제한 기간만 보관한다.

## 쟁점

### ISSUE-001 인증자료 저장/폐기 정책

상태: 제품/운영 초안 작성, 개발/보안 리뷰 요청

현재 문서는 "인증 원본은 심사 후 파기" 원칙을 갖고 있으나, 구현 가능한 운영 정책으로는 아직 부족하다.

결정이 필요한 항목:

- 업로드 원본의 최대 보관 기간
- 승인, 반려, 사용자 철회, 계정 삭제 시 각각의 폐기 시점
- 원본 파일과 파생 썸네일/미리보기의 동시 폐기 여부
- 파일 저장소 암호화 방식과 접근 권한
- 관리자 열람 기록, 다운로드 금지, 감사 로그 보관 기간
- 악성 파일 검사와 이미지 메타데이터 제거 정책

제안:

- 원본 인증자료는 별도 보안 버킷에 저장하고 DB에는 파일 ID, 심사 상태, 해시, 만료 시각만 둔다.
- 심사 완료 후 짧은 이의제기 기간이 지나면 원본을 자동 삭제한다.
- 장기 보관 대상은 인증 뱃지, 심사 결과, 반려 사유 코드, 감사 로그로 제한한다.

### ISSUE-002 DB에 저장하면 안 되는 민감정보

상태: 제품/운영 초안 작성, 개발/보안 리뷰 요청

프리미엄 검증 서비스 특성상 저장 금지 목록을 먼저 정해야 한다.

저장 금지 후보:

- 주민등록번호, 여권번호, 운전면허번호 원문
- 인증 서류 원본의 바이너리 또는 Base64 문자열
- 연락처 원문 목록
- 외부 SNS/OAuth access token 원문
- 결제 수단 전체 번호 또는 CVC
- AI 프롬프트에 그대로 들어간 민감 프로필 원문

제안:

- 연락처 차단은 서버 pepper 기반 해시로 처리하고 원문 연락처는 저장하지 않는다.
- 외부 인증 결과는 provider reference, 검증 상태, 발급 시각, 만료 시각만 저장한다.
- AI 분석에는 사용자 식별자와 인증 원본을 보내지 않고, 최소화된 feature와 익명 후보 ID만 보낸다.

### ISSUE-003 추천/매칭 API 경계

상태: 미합의

현재 문서는 추천 흐름은 있으나 API 책임 경계가 아직 약하다.

초안 경계:

- `Auth API`: 로그인, 토큰 갱신, 본인 인증 상태 조회
- `Profile API`: 기본 프로필, 사진, 선호 조건, 공개 범위
- `Verification API`: 셀렉트 신청, 증빙 업로드 URL 발급, 심사 상태 조회
- `Matching API`: 후보군 생성, 카드 조회, 수락/거절, 숨김
- `Recommendation API`: AI 설명, 궁합 요약, 관계 리스크 문구 생성
- `Safety API`: 신고, 차단, 지인 차단, 노출 제한
- `Admin API`: 심사 큐, 승인/반려, 큐레이션, 감사 로그

제안:

- 후보군 생성과 AI 설명 생성을 분리한다.
- AI는 최종 권한 판단자가 아니라 설명/요약/운영자 보조 역할로 제한한다.
- 추천 결과에는 `reason`, `caution`, `confidence`, `source_version`을 남겨 운영자가 재검토할 수 있게 한다.

### ISSUE-004 데이터 모델 초안

상태: 미합의

Asteria의 기존 모델을 참고하되 새 서비스의 비대칭 노출 정책에 맞춘 최소 모델이 필요하다.

초안 테이블:

- `users`
- `user_profiles`
- `user_photos`
- `select_opt_ins`
- `verification_applications`
- `verification_evidence`
- `verification_reviews`
- `verification_badges`
- `match_candidates`
- `daily_recommendations`
- `recommendation_explanations`
- `match_actions`
- `user_blocks`
- `contact_hash_blocks`
- `reports`
- `admin_users`
- `admin_audit_logs`

주의:

- `verification_evidence`는 원본 자체가 아니라 보안 저장소 참조와 폐기 상태만 가져야 한다.
- `recommendation_explanations`에는 민감 원문 대신 생성 결과, 모델/프롬프트 버전, 근거 feature 버전만 둔다.
- `admin_audit_logs`는 승인/반려/열람/권한 변경을 빠짐없이 기록해야 한다.

### ISSUE-005 사용자 플로우/화면 명세

상태: 미합의

문서는 사용자 여정을 잘 설명하지만, 구현 착수에는 화면 단위 명세가 더 필요하다.

필요한 화면:

- 가입/로그인
- 기본 프로필 작성
- 사진 등록
- 성향/가치관 설문
- 셀렉트 추천 opt-in 설정
- 셀렉트 검증 신청
- 심사 대기/반려/재제출
- 추천 카드
- 추천 이유 상세
- 수락/거절/숨김
- 신고/차단
- 내 정보/노출 설정
- 관리자 심사 큐
- 관리자 회원 상세
- 관리자 큐레이션/추천 조정

제안:

- 각 화면에 필수 입력값, 빈 상태, 오류 상태, 권한 부족 상태, 철회/삭제 동선을 함께 적는다.
- opt-in 문구는 일반 회원의 노출 범위와 철회 가능성을 명확히 고지해야 한다.

### ISSUE-006 React + Vite + Capacitor 구현 리스크

상태: 미합의

스택 방향은 타당하지만 WebView 앱 특유의 리스크를 구현 기준으로 더 명확히 해야 한다.

리스크:

- iOS/Android 권한 프롬프트 문구와 앱 심사 이슈
- 카메라/파일 업로드 실패 시 재시도와 부분 제출 상태
- WebView 로컬 저장소에 민감정보가 남는 문제
- 푸시 토큰과 세션 토큰의 저장 위치 혼동
- 딥링크/앱링크 인증 흐름 충돌
- 캡처 방지 정책의 플랫폼별 한계
- 네트워크 불안정 상태에서 추천 카드 액션 중복 전송

제안:

- 인증자료, 세션 토큰, 푸시 토큰 저장소를 명확히 분리한다.
- 모든 수락/거절/심사 액션에는 idempotency key를 둔다.
- Capacitor 플러그인은 화면 컴포넌트에서 직접 호출하지 않고 `src/platform/` 어댑터와 feature service를 거친다.

### ISSUE-007 관리자 심사/운영 도구 리스크

상태: 미합의

관리자 도구가 서비스 신뢰의 핵심이므로 일반 CRUD 대시보드 이상으로 설계해야 한다.

리스크:

- 과도한 관리자 권한
- 인증자료 열람 남용
- 승인/반려 기준의 일관성 부족
- 운영자 큐레이션이 차별적 노출로 보일 위험
- 위험 엔드포인트가 테스트/운영에 함께 남는 문제
- 감사 로그 수정 또는 누락

제안:

- 관리자 권한을 심사자, 운영자, 보안 관리자, 슈퍼 관리자 역할로 나눈다.
- 인증자료 열람은 사유 입력과 감사 로그를 강제한다.
- 승인/반려는 사유 코드와 자유 메모를 분리한다.
- 운영자 수동 추천은 사용자에게 노출되는 설명과 내부 근거를 분리해 저장한다.

### ISSUE-008 AI 추천 설명의 정책 경계

상태: 미합의

AI 추천 이유는 서비스 차별점이지만, 확정적 판단이나 차별적 표현으로 보이면 위험하다.

제안:

- 설명은 "가능성", "경향", "초반 조율 포인트" 중심으로 작성한다.
- 소득, 직업, 외모, 학력은 직접적 우열 표현에 쓰지 않는다.
- AI 설명에는 검증자료 원본이나 민감 식별정보를 포함하지 않는다.
- 사용자가 설명 삭제 또는 재생성을 요청할 수 있는 정책을 정한다.
