# 인증자료 수집/검증/보관 명세서

상태: 제품/운영 초안
작성일: 2026-05-21
기준 문서: `docs/privacy-verification-data-policy.md`, `docs/select-premium-verification-criteria.md`, `docs/member-tier-policy.md`
대상 서비스: 프리미엄 검증 기반 AI 소개팅 서비스

## 1. 목적

이 문서는 셀렉트/프리미엄 검증 과정에서 사용자가 제출하는 인증자료를 어떤 절차로 수집, 전처리, 열람, 심사, 보관, 삭제할지 정의한다.

핵심 원칙은 원본 인증자료를 검증 목적에 필요한 기간만 제한 보관하고, 서비스 DB에는 원본 파일, 고유식별번호 원문, OCR 원문 전체를 저장하지 않는 것이다.

## 2. 적용 범위

이 문서는 다음 자료에 적용한다.

- 재직, 경력, 학력, 전문 자격 증빙
- 소득, 자산, 사업자, 임원 지위 증빙
- 공개 영향력 확인 자료
- 운영 추천 또는 초대 검증을 뒷받침하는 외부 평판 자료
- 검증 신청 과정에서 사용자가 추가 제출한 보완 자료

이 문서는 실제 국가별 법률 검토 전 제품/운영 초안이다. 실제 출시 전에는 개인정보보호, 신용정보, 금융정보, 전자상거래, 위치정보, 결제 관련 의무 보존 기간과 충돌하지 않게 법무 검토가 필요하다.

## 3. 사용자 고지와 동의

검증 신청 화면은 업로드 전에 다음 내용을 고지해야 한다.

| 고지 항목 | 사용자 표시 기준 |
|---|---|
| 수집 목적 | 셀렉트/프리미엄 검증 심사와 부정 제출 방지 목적 |
| 제출 자료 | 선택한 검증 유형별 제출 가능 자료 예시 |
| 열람 주체 | 권한 있는 심사자, 보안 관리자, 필요한 경우 슈퍼 관리자 |
| 보관 기간 | 심사 대기, 승인, 반려, 철회별 원본 보관 기간 |
| 심사 후 잔존 정보 | 검증 유형, 상태, 뱃지, 반려 사유 코드, 감사 로그 |
| 삭제 기준 | 승인/반려/철회/계정 삭제 시 삭제 대기열 진입 기준 |
| AI 사용 제한 | 인증자료 원본과 OCR 원문은 AI 추천에 사용하지 않음 |
| 사용자 권리 | 신청 철회, 재제출, 계정 삭제, 문의 가능 경로 |

동의는 일반 약관 동의와 분리한다. 사용자가 동의하지 않으면 검증 신청을 시작할 수 없다.

## 4. 제출 자료 유형

| 검증 유형 | 제출 가능 자료 예시 | 수집 최소화 기준 |
|---|---|---|
| 주요 기업/기관 재직 | 재직증명서, 사내 이메일 확인, 명함, 직원 포털 일부 가림 캡처 | 회사명, 재직 상태, 발급/확인 시점 중심 |
| 고소득 | 소득금액증명, 원천징수영수증, 급여명세 일부 가림본 | 기준 충족 여부와 기간만 확인 |
| 고자산 | 잔고증명, 평가서, 등기/보유 증빙 일부 가림본 | 본인 소유, 기준 시점, 기준 충족 여부만 확인 |
| 전문 자격 | 자격증, 면허증, 협회/기관 조회 결과 | 자격명, 유효 상태, 본인 여부 중심 |
| 학력/경력 | 졸업증명, 재학/수료 증명, 경력증명 | 기관명, 상태, 기간 중심 |
| 창업/임원/사업자 | 사업자등록, 법인등기, 임원 증빙, 언론/공시 자료 | 사업 상태, 지위, 본인 여부 중심 |
| 공개 영향력 | 공개 SNS/채널, 언론 기사, 플랫폼 인증, 팔로워/조회 지표 | 공개 URL, 지속성, 본인 확인 중심 |
| 초대/운영 추천 | 기존 회원 추천, 운영자 인터뷰, 외부 평판 확인 | 추천 근거와 이해관계 고지 중심 |

사용자에게는 불필요한 고유식별번호, 계좌 전체 번호, 카드 전체 번호, 제3자 개인정보를 가리고 제출하도록 안내한다.

## 5. 상태 정의

### 사용자 표시 상태

| 상태 | 의미 | 사용자 행동 |
|---|---|---|
| 미신청 | 검증 신청을 시작하지 않음 | 신청 가능 |
| 작성 중 | 자료 선택 또는 업로드 전 | 저장 후 계속 또는 취소 |
| 제출 완료 | 자료 제출 완료, 심사 전 | 철회 가능 |
| 심사 중 | 심사자가 검토 중 | 대기 |
| 재제출 요청 | 자료 부족 또는 보완 필요 | 보완 제출 |
| 승인 | 검증 완료 | 유효기간 확인 |
| 반려 | 기준 미충족 또는 리스크 확인 | 사유 확인, 재신청 가능 여부 확인 |
| 철회 완료 | 사용자가 신청을 철회함 | 새 신청 가능 |
| 만료 | 유효기간 경과 | 재검증 가능 |

### 운영자 내부 상태

| 상태 | 의미 |
|---|---|
| draft | 신청 작성 중 |
| submitted | 제출 완료 |
| preprocessing | 자동 전처리 중 |
| preprocessing_failed | 전처리 실패 |
| queued | 심사 큐 대기 |
| in_review | 심사자 검토 중 |
| needs_resubmission | 보완 요청 |
| approved | 승인 |
| rejected | 반려 |
| withdrawn | 사용자 철회 |
| expired | 유효기간 만료 |
| deletion_queued | 원본 삭제 대기 |
| deleted | 원본 삭제 완료 |
| deletion_failed | 삭제 실패, 재시도 필요 |

사용자 표시 상태와 내부 상태는 분리한다. 내부 상태 이름, 저장소 객체 ID, 삭제 실패 세부 사유는 사용자에게 직접 노출하지 않는다.

## 6. 수명주기

### 6.1 신청 시작

- 사용자는 검증 유형을 선택한다.
- 서비스는 검증 유형별 제출 가능 자료와 가림 안내를 보여준다.
- 사용자가 자료 수집, 열람, 보관, 삭제 기준에 동의하면 신청을 시작한다.
- 제한, 탈퇴 예정, 중대한 미해결 신고 상태의 계정은 신청을 시작할 수 없다.

### 6.2 업로드

- 파일은 서버 또는 보안 저장소의 제한된 업로드 경로로 전송한다.
- 클라이언트는 인증자료를 WebView localStorage, IndexedDB, 일반 캐시에 저장하지 않는 것을 원칙으로 한다.
- 업로드 실패 시 사용자는 같은 신청에서 재시도할 수 있다.
- 업로드 완료 전에는 심사 큐에 들어가지 않는다.

### 6.3 자동 전처리

전처리 단계는 다음을 수행한다.

- 파일 형식과 크기 확인
- 악성 파일 검사
- 이미지 메타데이터 제거
- 미리보기 또는 썸네일이 필요한 경우 민감정보 노출 최소화
- OCR이 필요한 경우 전체 원문 저장 금지, 검증에 필요한 판정 결과만 저장

전처리 실패 시 사용자에게는 "파일을 확인할 수 없음", "지원하지 않는 형식", "다시 업로드 필요"처럼 제한된 사유만 표시한다.

### 6.4 심사 큐 진입

- 전처리가 성공한 신청은 심사 큐에 들어간다.
- 심사 우선순위는 제출 시각, 재제출 여부, 운영 이벤트를 기준으로 정한다.
- 심사 큐에는 계정 안전 리스크, 신고/제재 상태, 이전 반려 이력을 함께 표시할 수 있다.
- 인증자료 원본은 심사자가 열람 사유를 입력한 뒤에만 열람할 수 있다.

### 6.5 관리자 열람

심사자가 인증자료를 열람하기 전에는 다음을 기록한다.

- 열람자 ID
- 대상 사용자 ID
- 신청 ID
- 열람 사유 코드
- 자유 사유가 필요한 경우 제한된 메모
- 열람 시각
- 요청 IP 또는 기기 정보

다운로드는 기본 금지한다. 화면 캡처 방지는 플랫폼 한계가 있으므로 보조 수단으로만 본다.

### 6.6 승인

승인 시 저장하는 정보는 다음으로 제한한다.

- 신청 ID
- 사용자 ID
- 검증 유형
- 승인 상태
- 검증 뱃지
- 승인 사유 코드
- 심사자 ID
- 승인 시각
- 유효기간
- 재검증 예정 시각
- 원본 삭제 예정 시각

승인 후 원본 인증자료는 `docs/privacy-verification-data-policy.md`의 초안 기준에 따라 승인 후 7일 뒤 삭제되도록 삭제 대기열에 넣는다.

### 6.7 재제출 요청

재제출 요청은 자료가 일부 부족하거나 확인이 불가능하지만 반려까지는 아닌 경우에 사용한다.

사용자에게 표시할 수 있는 사유:

- 자료가 흐리거나 일부 정보 확인이 어려움
- 발급일 또는 유효기간 확인이 필요함
- 신청한 검증 유형과 자료가 맞지 않음
- 불필요한 민감정보가 과도하게 노출되어 다시 제출 필요
- 본인 여부 확인이 추가로 필요함

재제출 시 이전 원본을 재사용하지 않고 새 자료를 받는다. 이전 원본은 정책상 보관 기간에 따라 삭제 대기열에 넣는다.

### 6.8 반려

반려는 기준 미충족, 본인 여부 불명확, 조작 의심, 안전 리스크, 중대한 정책 위반이 확인된 경우에 사용한다.

반려 후 저장하는 정보:

- 반려 상태
- 사용자 표시용 반려 사유 코드
- 재신청 가능 여부
- 심사자 ID
- 반려 시각
- 내부 검토 메모
- 원본 삭제 예정 시각

반려 후 원본 인증자료는 반려 후 14일 뒤 삭제되도록 삭제 대기열에 넣는다.

### 6.9 사용자 철회

사용자는 심사 전 또는 심사 중 신청을 철회할 수 있어야 한다.

철회 시 처리:

- 신청 상태를 철회로 변경한다.
- 아직 심사 큐에 있으면 큐에서 제거한다.
- 원본과 파생 파일은 24시간 이내 삭제 대기열에 넣는다.
- 이미 생성된 감사 로그와 행위 기록은 민감 원문 없이 보존한다.

### 6.10 계정 삭제

계정 삭제 요청이 들어오면 법적/운영상 보존 대상이 아닌 원본 인증자료와 파생 파일을 삭제 대기열에 넣는다.

검증 뱃지, 심사 결과, 감사 로그, 신고/제재 기록은 법무/운영 보존 기준에 따라 별도 처리한다. 단, 원본 파일이나 고유식별번호 원문은 장기 보관하지 않는다.

## 7. 보관 기간

| 대상 | 최대 보관 기간 초안 | 삭제 기준 |
|---|---:|---|
| 심사 대기 원본 | 30일 | 장기 미완료 신청 자동 만료 후 삭제 대기열 |
| 승인 후 원본 | 7일 | 이의제기 대응 기간 후 삭제 대기열 |
| 반려 후 원본 | 14일 | 재심사 요청 가능 기간 후 삭제 대기열 |
| 철회 원본 | 24시간 이내 삭제 대기열 | 사용자 철회 즉시 삭제 예약 |
| 전처리 실패 파일 | 7일 이내 | 사용자 재업로드 기회 후 삭제 |
| 썸네일/미리보기 | 원본과 동일하거나 더 짧게 | 원본 삭제 시 함께 삭제 |
| 심사 결과/뱃지 | 회원 유지 기간 | 원본 없이 결과와 상태만 보관 |
| 관리자 감사 로그 | 최소 3년 초안 | 민감 원문 없이 행위 기록만 보관 |

보관 기간은 법무 검토 전 초안이다.

## 8. 삭제와 재시도

삭제 대기열은 다음 상태를 가져야 한다.

| 상태 | 의미 |
|---|---|
| queued | 삭제 예약됨 |
| processing | 삭제 작업 중 |
| deleted | 삭제 완료 |
| retry_scheduled | 실패 후 재시도 예약 |
| failed_requires_review | 반복 실패로 보안 관리자 확인 필요 |

삭제 실패 시 사용자에게 원본 파일명, 저장소 경로, 내부 오류를 노출하지 않는다. 운영자는 실패 원인, 재시도 횟수, 마지막 시각, 담당 작업자 또는 배치 ID를 확인할 수 있어야 한다.

## 9. 감사 로그 이벤트

최소 감사 로그 이벤트는 다음과 같다.

- verification_application_started
- verification_evidence_uploaded
- verification_evidence_preprocessed
- verification_evidence_viewed
- verification_review_started
- verification_resubmission_requested
- verification_approved
- verification_rejected
- verification_withdrawn
- verification_expired
- verification_evidence_deletion_queued
- verification_evidence_deleted
- verification_evidence_deletion_failed
- verification_badge_granted
- verification_badge_revoked

감사 로그에는 인증자료 원본, OCR 원문, 고유식별번호 원문을 포함하지 않는다.

## 10. 사용자 문구 기준

사용자에게는 다음 표현을 피한다.

- 상위 회원으로 인정되었습니다
- 고급 회원으로 승격되었습니다
- 일반 회원보다 우수합니다
- 자산/소득 수준이 뛰어납니다

권장 표현:

- 검증이 완료되었습니다
- 재직 정보가 확인되었습니다
- 전문 자격이 확인되었습니다
- 자료 보완이 필요합니다
- 검증 기준을 충족하지 못했습니다

1. `verification_applications`, `verification_evidence`, `verification_reviews` 상태 전이가 구현 가능한지
2. 원본, 썸네일, 미리보기, 전처리 실패 파일을 삭제 대기열로 함께 관리하는 방식
3. 승인 후 7일, 반려 후 14일, 철회 후 24시간 삭제 기준의 배치/재시도 설계
4. 열람 사유 입력과 감사 로그를 API/DB 레벨에서 강제하는 방법
5. OCR 결과를 원문 저장 없이 판정 결과만 남기는 안전한 구조
6. Capacitor 앱에서 업로드 파일이 로컬 저장소에 남지 않게 하는 구현 기준
