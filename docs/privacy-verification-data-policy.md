# 개인정보 및 인증자료 처리 정책 초안

상태: 제품/운영 초안  
협업 범위: `ISSUE-001 인증자료 저장/폐기 정책`, `ISSUE-002 DB에 저장하면 안 되는 민감정보`  
리뷰 요청 대상: 개발/구현/보안 담당

이 문서는 프리미엄 검증 기반 소개팅 서비스에서 개인정보와 인증자료를 어떻게 수집, 저장, 열람, 파기할지 정리하는 운영 기준 초안이다. 공개 저장소용 문서이므로 실제 사용자 데이터, 인증자료 원본, 내부 키, 로그 원문은 포함하지 않는다.

## 1. 기본 원칙

- 인증자료 원본은 장기 보관하지 않는다.
- DB에는 인증자료 원본이나 민감 식별번호 원문을 저장하지 않는다.
- 장기 보관 대상은 심사 결과, 인증 뱃지, 반려 사유 코드, 감사 로그로 제한한다.
- 관리자 열람은 역할, 목적, 사유, 시간, 대상이 감사 로그로 남아야 한다.
- AI 추천에는 인증자료 원본, 민감 식별정보, 사용자 실명 식별자를 직접 전달하지 않는다.
- 사용자는 인증 신청 철회, 추천 opt-in 철회, 계정 삭제를 요청할 수 있어야 한다.

## 2. 인증자료 수명주기

| 단계 | 처리 기준 | 보관 기준 |
|---|---|---|
| 업로드 | 사용자가 검증 신청 과정에서 자료를 제출한다. | 원본은 별도 보안 저장소에 임시 보관한다. |
| 자동 전처리 | 악성 파일 검사, 파일 형식 확인, 이미지 메타데이터 제거를 수행한다. | 전처리 결과와 실패 사유만 기록한다. |
| 관리자 심사 | 권한 있는 심사자가 열람 사유를 입력한 뒤 자료를 확인한다. | 열람 기록은 감사 로그로 남긴다. |
| 승인 | 검증 뱃지, 검증 유형, 심사 결과, 승인 시각을 저장한다. | 원본은 이의제기 가능 기간 후 삭제한다. |
| 반려 | 반려 사유 코드, 재제출 가능 여부, 반려 시각을 저장한다. | 원본은 이의제기 가능 기간 후 삭제한다. |
| 재심사 | 필요한 경우 새 자료를 다시 제출받는다. | 이전 원본은 재사용하지 않는다. |
| 사용자 철회 | 사용자가 검증 신청을 철회한다. | 원본은 삭제 대기열에 넣고 빠르게 파기한다. |
| 계정 삭제 | 사용자가 계정 삭제를 요청한다. | 법적/운영상 보존 대상이 아닌 원본과 프로필 자료를 파기한다. |

## 3. 보관 기간 초안

| 항목 | 최대 보관 기간 초안 | 비고 |
|---|---:|---|
| 심사 대기 중 인증자료 원본 | 30일 | 장기 미완료 신청은 자동 만료 처리한다. |
| 승인 후 인증자료 원본 | 승인 후 7일 | 이의제기 대응 기간 후 자동 삭제한다. |
| 반려 후 인증자료 원본 | 반려 후 14일 | 재심사 요청 가능 기간 후 자동 삭제한다. |
| 사용자 철회 자료 | 철회 후 24시간 이내 삭제 대기열 | 운영 장애 시 재시도 로그만 남긴다. |
| 인증 뱃지/심사 결과 | 회원 유지 기간 동안 | 원본 없이 결과와 상태만 보관한다. |
| 반려 사유 코드 | 회원 유지 기간 또는 운영 보존 기간 | 차후 재심사 일관성 확인용이다. |
| 관리자 감사 로그 | 최소 3년 초안 | 민감 원문 없이 행위 기록만 보관한다. |
| 신고/제재 기록 | 운영 정책에 따른 별도 기간 | 안전 대응 목적의 최소 정보만 보관한다. |

보관 기간은 법무 검토 전 초안이다. 실제 출시 전에는 국가별 개인정보보호법, 전자상거래, 위치정보, 결제 관련 의무 보존 기간과 충돌하지 않게 조정해야 한다.

## 4. DB 저장 허용 항목

DB에는 다음처럼 서비스 운영에 필요한 최소 정보만 저장한다.

| 영역 | 저장 가능 항목 |
|---|---|
| 사용자 | 내부 사용자 ID, 가입 상태, 권한, 약관 동의 버전, 계정 상태 |
| 프로필 | 닉네임, 나이대, 지역 단위, 선호 조건, 공개 범위, 프로필 완성도 |
| 검증 신청 | 신청 ID, 사용자 ID, 검증 유형, 신청 상태, 신청/만료 시각 |
| 인증자료 참조 | 보안 저장소 객체 ID, 파일 유형, 업로드 상태, 삭제 예정 시각, 삭제 상태 |
| 심사 결과 | 승인/반려 상태, 검증 뱃지, 반려 사유 코드, 심사자 ID, 심사 시각 |
| 감사 로그 | 관리자 ID, 대상 사용자 ID, 행위 유형, 열람 사유, 요청 IP/기기 정보, 시각 |
| AI 추천 | 익명 후보 ID, feature 버전, 추천 이유 결과, 모델/프롬프트 버전 |

## 5. DB 저장 금지 항목

다음 정보는 DB에 원문으로 저장하지 않는다.

- 주민등록번호, 여권번호, 운전면허번호, 외국인등록번호 원문
- 신분증, 재직증명서, 소득증빙, 자산증빙, 학력증빙 원본 파일
- 인증 서류 이미지의 Base64 문자열
- OCR로 추출한 인증자료 원문 전체
- 결제 카드 전체 번호, CVC, 계좌 비밀번호
- 외부 OAuth access token, refresh token 원문
- 연락처 주소록 원문 목록
- 차단용 연락처 원문
- AI 프롬프트에 그대로 포함된 민감 프로필 원문
- 내부 운영자 메모 중 차별적 표현, 민감 추정, 불필요한 사생활 정보

## 6. 연락처/지인 차단 처리

- 연락처 기반 지인 차단은 원문 연락처를 저장하지 않는다.
- 서버에서 정규화한 연락처에 pepper 기반 해시를 적용해 비교한다.
- pepper는 애플리케이션 DB와 분리된 시크릿 저장소에서 관리한다.
- 사용자가 연락처 동의를 철회하면 새 비교를 중단하고, 기존 해시 처리 데이터의 삭제 정책을 제공한다.

## 7. AI 추천 입력 제한

AI 추천에는 다음 정보만 전달한다.

- 익명화된 후보 ID
- 연령대, 지역 단위, 선호 조건처럼 매칭에 필요한 최소 feature
- 검증 원본이 아닌 검증 상태와 뱃지 정보
- 성향 설문에서 파생된 요약 feature
- 추천 설명 생성에 필요한 비식별 요약값

AI 추천에는 다음 정보를 전달하지 않는다.

- 실명
- 전화번호, 이메일, 주소
- 인증자료 원본 또는 OCR 원문
- 주민등록번호 등 고유식별정보
- 결제 정보
- 관리자 내부 메모 원문

## 8. 관리자 열람 정책

- 인증자료 열람은 심사자, 보안 관리자, 슈퍼 관리자 등 제한된 역할만 가능하다.
- 인증자료 열람 전에는 열람 사유를 입력해야 한다.
- 다운로드는 기본 금지한다.
- 화면 캡처 방지는 플랫폼 한계가 있으므로 보조 수단으로만 본다.
- 모든 열람, 승인, 반려, 재심사 요청, 권한 변경은 감사 로그로 남긴다.
- 감사 로그는 운영자가 수정할 수 없고, 별도 권한으로만 조회한다.

## 9. 사용자 고지와 동의

검증 신청 화면에는 최소한 다음 내용을 고지한다.

- 어떤 자료를 제출하는지
- 어떤 목적으로 사용하는지
- 누가 열람하는지
- 원본을 얼마나 보관하는지
- 심사 후 어떤 정보만 남는지
- 신청 철회와 계정 삭제 시 어떻게 파기되는지
- AI 추천에 인증자료 원본을 사용하지 않는다는 점

일반 회원의 셀렉트 추천 opt-in 화면에는 다음을 명확히 고지한다.

- 일반 회원은 셀렉트 회원 목록을 직접 볼 수 없다.
- opt-in한 경우에만 셀렉트 회원에게 추천 후보로 포함될 수 있다.
- opt-in은 언제든 철회할 수 있다.
- 철회 후에는 신규 추천 후보에 포함되지 않는다.

## 10. 개발/구현 담당 리뷰 요청

개발/구현/보안 관점에서 다음을 검토해야 한다.

1. 위 보관 기간이 실제 배치 작업과 삭제 대기열로 구현 가능한지
2. `verification_evidence` 테이블에 어떤 참조 필드만 남겨야 하는지
3. 파일 삭제 실패, 재시도, 삭제 증적을 어떻게 기록할지
4. 관리자 열람 로그를 변조 불가능하게 남기는 방법
5. 연락처 해시 비교에서 pepper 관리와 회전 정책
6. AI 추천 feature 생성 시 비식별화 경계
7. Capacitor 앱에서 인증자료가 로컬 캐시나 WebView storage에 남지 않게 하는 방법
8. 공개 저장소 문서 수준에서 더 숨겨야 하는 구현 세부사항
