# 프리미엄 검증 기반 AI 소개팅 서비스 기획 정리

## 1. 한 줄 정의

AI가 사람의 성향, 가치관, 대화 방식, 생활 맥락을 분석해서 잘 어울리는 상대를 추천하고, 추천 이유까지 설명해주는 프리미엄 소개팅 서비스.

## 2. 핵심 문제

기존 소개팅 앱은 보통 사진, 나이, 지역, 직업, 키워드 같은 얕은 정보에 의존한다. 그 결과 사용자는 많은 프로필을 직접 훑어야 하고, 실제로 왜 잘 맞는지 알기 어렵다.

이 서비스는 단순 노출형 앱이 아니라, 검증된 회원 풀과 AI 분석을 기반으로 "소개받는 느낌"에 가까운 신뢰형 매칭 경험을 만든다.

## 3. 핵심 가치

- AI가 사용자를 정밀 분석해 궁합 가능성이 높은 상대를 추천한다.
- 추천 결과에는 매칭 이유를 함께 제공한다.
- 일반 회원과 프리미엄/셀렉트 회원을 분리해 신뢰와 희소성을 유지한다.
- 프리미엄 회원은 단순 본인 인증이 아니라 사회적 신뢰도와 자격을 강하게 검증한다.
- 일반 회원은 직접 프리미엄 풀을 조회하지 못하고, 본인이 동의한 경우에만 프리미엄 회원에게 추천될 수 있다.

## 4. 회원 구조

### 일반 회원

기본 가입 회원이다. 프로필, 관심사, 성향 질문, 대화 스타일 분석 등을 통해 AI 매칭 대상이 된다.

일반 회원은 프리미엄/셀렉트 회원의 프로필을 직접 열람할 수 없다. 다만 본인이 동의하면 "프리미엄 회원에게 추천 가능" 상태가 될 수 있다.

### 인기 일반 회원

일반 회원 중에서 프로필 완성도, 응답률, 신고 이력, 매칭 후 피드백, 선호도 지표가 좋은 회원이다.

인기 일반 회원은 프리미엄 회원에게 추천될 가능성이 높아질 수 있지만, 이 역시 본인의 동의와 노출 설정을 전제로 한다.

### 프리미엄/셀렉트 회원

단순히 얼굴이나 실명 인증을 완료한 회원이 아니다. 더 강한 자격 검증을 통과한 회원이다.

검증 예시는 다음과 같다.

- 대기업, 전문직, 공공기관, 스타트업 핵심 직무 등 직업 검증
- 고소득 또는 자산 기준 검증
- 의사, 변호사, 회계사, 교수, 연구원 등 전문 자격 검증
- 인플루언서, 크리에이터, 유명인 등 공개 영향력 검증
- 학력, 경력, 수상, 포트폴리오 등 추가 신뢰 자료 검증

프리미엄 회원끼리는 별도 큐레이션 풀에서 매칭될 수 있고, 프리미엄 회원은 선택적으로 검증된 일반/인기 일반 회원 추천을 받을 수 있다.

## 5. AI 정밀 분석 매칭

AI 정밀 분석 매칭은 전체 매칭 기준 중 하나이지만, 서비스의 차별화 핵심이다. 단순 키워드 일치가 아니라 사람의 성향과 관계 가능성을 다층적으로 분석한다.

### 분석 축

- 심리/성향 분석: 성격, 애착 유형, 감정 표현 방식, 갈등 대응 방식
- 가치관 분석: 결혼관, 일과 삶의 균형, 가족관, 소비 성향, 성장 욕구
- 대화 방식 분석: 말투, 답변 길이, 질문 성향, 유머 코드, 공감 방식
- 생활 맥락 분석: 직업 리듬, 활동 지역, 취미, 주말 패턴, 장기 목표
- 선호 조건 분석: 외형 조건뿐 아니라 관계에서 중요하게 보는 기준

### 매칭 결과

AI는 단순 점수만 보여주지 않고, 사람이 납득할 수 있는 설명을 제공한다.

예시 설명 구조:

- "두 사람 모두 장기적 안정성을 중요하게 봅니다."
- "A는 계획형이고 B는 실행력이 높아 생활 리듬이 잘 맞을 가능성이 있습니다."
- "갈등 상황에서 둘 다 직접 대화를 선호해 오해가 길어질 가능성이 낮습니다."
- "다만 여가 방식은 다르므로 초반에는 활동 선택을 조율하는 것이 좋습니다."

## 6. 추천 흐름

1. 가입 및 기본 인증
2. 프로필 작성
3. 성향/가치관 질문 응답
4. AI 정밀 분석
5. 회원 등급 및 노출 권한 설정
6. 후보군 필터링
7. AI 매칭 스코어 산출
8. 운영자 또는 시스템 큐레이션
9. 추천 카드 제공
10. 수락, 거절, 대화, 피드백 수집
11. 피드백 기반 매칭 개선

## 7. 노출 정책

프리미엄 풀의 희소성과 안전성을 유지하려면 노출 정책이 중요하다.

- 일반 회원은 프리미엄 회원 목록을 직접 볼 수 없다.
- 프리미엄 회원 프로필은 일반 탐색 화면에 노출하지 않는다.
- 일반 회원은 "프리미엄 회원에게 추천 허용"을 직접 켜야 한다.
- 프리미엄 회원은 추천받을 일반 회원의 범위를 선택할 수 있다.
- 프리미엄 회원끼리의 매칭은 별도 큐레이션으로 운영한다.
- 모든 추천에는 거절권, 숨김, 신고, 재추천 제한이 있어야 한다.

## 8. MVP 범위

초기 버전은 기능을 과하게 넓히기보다 신뢰와 매칭 설명의 품질을 검증하는 데 집중한다.

필수 기능:

- 회원 가입 및 기본 프로필
- 일반/프리미엄 등급 구조
- 프리미엄 신청 및 운영자 검증 워크플로우
- AI 성향 설문
- AI 매칭 점수와 추천 이유 생성
- 추천 카드
- 수락/거절
- 피드백 수집

초기에는 완전 자동 매칭보다 운영자 큐레이션을 섞는 방식이 안전하다. 프리미엄 회원의 신뢰도가 서비스 핵심이므로, 자동화보다 품질 관리가 우선이다.

## 9. 구현 방향 메모

사용자 경험은 빠르게 실험할 수 있어야 한다. 초기 프로토타입은 HTML/CSS/JS 기반 웹으로 만들고, 이후 Capacitor로 모바일/데스크톱 앱까지 확장하는 방식이 적합하다.

권장 방향:

- React + Vite + TypeScript 기반의 하나의 웹 코드베이스를 먼저 만든다.
- 프로필/설문/추천 카드 흐름을 HTML 기반으로 빠르게 검증한다.
- 모바일 앱 전환은 Capacitor로 감싼다.
- AI 분석은 초기에는 서버 API로 분리한다.
- 실제 민감 정보와 검증 자료는 별도 보안 저장소에 둔다.
- 화면 전환, 모달, 추천 카드 애니메이션은 `transform`과 `opacity`만 사용한다.
- 복잡한 모바일 제스처와 전환은 `Framer Motion`으로 구현한다.
- 카메라, 푸시, 로컬 저장소, 파일 접근은 Capacitor 공식 플러그인을 통해서만 호출한다.

## 10. Asteria 분석 반영

기존 앱 `Asteria`는 이미 프리미엄 소개팅 서비스의 여러 골격을 갖고 있다.

- 시간대별 데일리 카드 추천 구조
- 능력/외모 인증과 관리자 심사 흐름
- 인증 뱃지 기반 추천/필터링
- 골드, 쿠폰, 프로필 해금, 조건 추천 수익 구조
- 차단, 연락처 기반 지인 차단, 신고/문의/공지 흐름
- ASP.NET Core API, MariaDB, SignalR, 관리자 웹 대시보드 구조

새 기획에서는 이 구조를 그대로 복제하기보다 다음 방향으로 정리한다.

- `프리미엄`을 단순 구독자가 아니라 `셀렉트 검증 회원`으로 재정의한다.
- 일반 회원이 셀렉트 회원 목록을 직접 조회하지 못하게 한다.
- 일반 회원은 opt-in한 경우에만 셀렉트 회원에게 추천될 수 있게 한다.
- AI는 자동 매칭보다 추천 이유, 궁합 근거, 관계 리스크 설명부터 담당한다.
- 운영자는 초기에 큐레이션 품질을 보정하고, AI는 운영자의 판단을 보조한다.
- 인증 원본은 심사 후 파기하고, 장기 보관 대상은 뱃지, 심사 결과, 감사 로그로 제한한다.

상세 분석은 [Asteria 역기획서](asteria-reverse-engineering.md)에 정리했다.

구현 방향은 Asteria의 Flutter/Blazor 구조를 그대로 이어가지 않고, `React + Vite + Capacitor`로 재정렬한다. 이 방식은 웹, iOS, Android를 하나의 코드베이스로 묶으면서도 표준 HTML DOM을 유지해 텍스트 선택, 스크래핑, 외부 도구 주석 작업을 가능하게 한다.

## 11. Asteria 기반 MVP 재정의

초기 MVP는 "기술을 다 만들기"보다 "검증된 사람을 안전하게 소개받는 경험"을 검증해야 한다.

1차 MVP 범위:

- 가입, 기본 프로필, 사진 등록
- 일반 회원의 셀렉트 추천 opt-in 설정
- 셀렉트 검증 신청과 관리자 심사
- 인증 뱃지와 프로필 신뢰 신호
- 시간대별 또는 운영자 큐레이션 추천 카드
- AI 추천 이유와 주의할 차이점 설명
- 수락, 거절, 피드백 수집
- 지인 차단, 신고, 숨김

Asteria의 골드/쿠폰/아이템 구조는 2차 이후로 미루고, 초기에는 검증 심사, 셀렉트 멤버십, AI 분석 리포트 중심으로 수익성을 검증한다.

## 12. 리스크와 주의점

- 검증 기준이 차별적으로 보이지 않도록 표현과 운영 정책을 조심해야 한다.
- 소득, 직업, 자산, 학력 같은 민감한 기준은 사용자 동의와 명확한 목적 고지가 필요하다.
- AI 매칭 설명은 확정적 판단이 아니라 가능성 중심으로 표현해야 한다.
- 외모, 소득, 직업만 강화하면 신뢰형 서비스가 아니라 계층형 노출 앱으로 보일 수 있다.
- 프리미엄 회원 보호와 일반 회원의 자존감 보호를 동시에 고려해야 한다.
- 공개 문서에는 실제 인증 자료, 사용자 데이터, 내부 로그, API 키, DB 접속 문자열을 절대 포함하지 않는다.
- 개발/테스트용 우회 계정, mock 데이터, 위험한 관리자 엔드포인트는 운영 전 제거해야 한다.
